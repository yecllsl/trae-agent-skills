# 协作链路

> **打开时机**：需要了解"这一步做完下一步干什么"时打开。—— **流程型操作手册**。方法论决策请查阅 [Skills协作.md](./Skills协作.md)。

本文档定义智能体协作流程、准入准出规则和异常处理机制。

***

## 1. 协作链路总览

### 1.1 项目Skill主链路

```
产品经理（产品规划）→ 
架构师（需求分析→架构设计→架构评审→任务拆解）→ 
开发工程师（功能开发→单元测试→本地验证）→ 
代码评审工程师（代码评审）→ 
测试工程师（测试策略→测试执行→Bug反馈）→ 
开发工程师（Bug修复）→ 
测试工程师（回归测试→上线结论）→ 
发布运维（文档更新→CICD验证→自动化发布→文档归档）
```

### 1.2 全局Skill嵌入链路

全局Skill以四种模式（串联/嵌入/守门/并行）贯穿项目Skill主链路。各Skill的触发条件、嵌入位置与模式详见 [Skills协作.md](./Skills协作.md)。

### 1.3 全景协作链路

```
                        ← [brainstorming] 需求不明确时前置
产品经理（产品规划）
        ↓
                        ← [brainstorming] 需求澄清
架构师（需求分析→架构设计→架构评审→任务拆解）
        ↓               ← [writing-plans] 任务拆解后编写实施计划
                        ← [using-git-worktrees] 创建隔离工作区
开发工程师（功能开发→单元测试→本地验证）
        ↓               ← [TDD] 编码过程内嵌
                        ← [subagent-driven-development] 单任务分步执行
                        ← [executing-plans] 独立会话批量执行
                        ← [dispatching-parallel-agents] 多任务并行
                        ← [verification] 每步完成验证
代码评审工程师（代码评审）
        ↓               ← [requesting-code-review] 可选快检
测试工程师（测试策略→测试执行→Bug反馈）
        ↓               
开发工程师（Bug修复）
        ↓               ← [systematic-debugging] Bug根因不明时前置
                        ← [TDD] 修复过程内嵌
                        ← [verification] 修复完成验证
                        ← [finishing-a-development-branch] 集成决策
测试工程师（回归测试→上线结论）
        ↓               ← [verification] 上线前验证
发布运维（文档更新→CICD验证→自动化发布→文档归档）
                        ← [verification] 最终守门
                        ← [git-pushing] 发布内嵌
```

## 2. 分阶段协作规范

### 阶段0：产品规划与项目管理（产品经理主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 | 嵌入全局Skill |
|------|------|------|------|---------|-------------|
| 产品经理 | PM-01 产品规划 | 市场调研、业务目标 | `/docs/product/产品规划方案.md` | 产品愿景清晰，路线图合理 | brainstorming（需求不明确时前置探索） |

### 阶段1：需求与架构（架构师主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 | 嵌入全局Skill |
|------|------|------|------|---------|-------------|
| 架构师 | ARC-01 需求分析 | 业务诉求 | `/docs/requirements/REQ_需求规格说明书.md` | 需求无歧义，验收标准可量化 | brainstorming（需求澄清与方案探索） |
| 架构师 | ARC-02 架构设计 | 需求规格 | `/docs/architecture/架构设计说明书.md` | 技术栈适配，接口规范完整 | — |
| 架构师 | ARC-03 任务拆解 | 架构方案（评审通过后） | `/docs/planning/task_list_*.md` | 任务粒度适配迭代周期 | writing-plans（任务拆解宏观切分后，微观展开实施计划） |
| 架构师 | ARC-04 架构评审 | 架构方案、需求规格 | `/docs/architecture/架构评审报告.md` | 技术可行性验证，风险识别完整 | verification-before-completion（评审结论需验证证据） |

### 阶段2：开发实现（开发工程师主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 | 嵌入全局Skill |
|------|------|------|------|---------|-------------|
| 开发工程师 | DEV-01 功能开发 | 任务清单、架构方案 | 代码 + `/tests/unit/` + `/docs/development/交付报告_*.md` | 覆盖率≥80%，测试通过率100% | using-git-worktrees（隔离工作区）+ TDD（编码内嵌）+ verification（每步验证）+ dispatching-parallel-agents（多任务并行时） |

### 阶段3：测试验证（测试工程师主导）

| 角色 | 指令 | 输入 | 输出 | 关键规则 | 嵌入全局Skill |
|------|------|------|------|---------|-------------|
| 测试工程师 | TST-01 测试策略 | 需求规格、架构方案 | `/docs/test/strategy_*.md` | 覆盖MVP需求 | — |

## 3. 全局Skill嵌入规范

全局Skill通过四种模式嵌入项目Skill主链路：

| 模式 | 说明 | 示例 |
|------|------|------|
| **串联** | 全局Skill输出作为项目Skill输入 | brainstorming → 产品规划 → 需求分析 |
| **嵌入** | 全局Skill作为方法论内嵌于项目Skill | TDD循环嵌入功能开发过程 |
| **守门** | 全局Skill作为质量关卡在关键节点拦截 | 功能开发 → [verification] → 代码评审 |
| **并行** | 全局Skill协调多个项目Skill并行执行 | dispatching-parallel-agents 调度多Agent |

> 全局Skill铁律、调用优先级、功能重叠决策速查 详见 [Skills协作.md](./Skills协作.md)。

- [ ] 产品规划方案已确认

### 4.2 开发准入

- [ ] 需求规格说明书已确认
- [ ] 架构设计方案已确认
- [ ] 架构评审通过（ARC-04）
- [ ] 任务清单已拆解
- [ ] [新增] 实施计划已编写（writing-plans）

### 4.3 测试准入

- [ ] 开发交付报告已提交
- [ ] 单元测试覆盖率≥80%
- [ ] 本地核心场景验证通过
- [ ] 代码评审通过（REV-01）
- [ ] [新增] 功能开发每步已完成verification验证

### 4.4 发布准入

- [ ] 测试上线结论为"建议上线"
- [ ] P0/P1级Bug 100%修复验证
- [ ] CICD流水线验证通过
- [ ] 版本号已更新
- [ ] 用户文档已更新（CHANGELOG、README、CLI使用指南、API文档、Agent配置指南、配置文件示例）
- [ ] 全局指路文档已更新（AGENTS.md、架构设计说明书、需求规格说明书、协作链路、指令手册）
- [ ] [新增] verification-before-completion最终验证已通过

### 4.5 发布准出

- [ ] 版本Tag创建成功
- [ ] GitHub Release发布成功
- [ ] 线上服务验证通过

- [ ] 项目目录已清理（docs/test/reports/、docs/devops/、docs/development/、docs/review/）

## 5. 异常处理

| 异常场景 | 处理方式 | 责任人 | 关联全局Skill |
|---------|---------|-------|-------------|
| 产品规划方向不明确 | 召开产品评审会议，重新定义产品愿景 | 产品经理 | brainstorming（开放性探索） |
| 需求理解存在歧义 | 逐个提问澄清，产出设计文档 | 架构师 | brainstorming（需求澄清） |
| 本地调试多次失败 | 输出调试报告，同步架构师排查 | 开发工程师 | systematic-debugging（根因定位） |
| Bug修复3次仍失败 | 停止修复，质疑架构设计 | 架构师 | systematic-debugging（Phase 3质疑架构） |
| CICD流水线连续失败 | 同步架构师、测试工程师共同分析 | 发布运维 | FIX-CI → systematic-debugging（配置型→直接修复；代码型→根因定位） |
| 架构评审不通过 | 按评审报告整改后重新提交 | 架构师 | — |
| 代码评审不通过 | 按评审报告整改后重新提交 | 开发工程师 | receiving-code-review（技术评估与实施） |
| 测试通过率<95% | 优先修复失败用例，达标后进入发布 | 开发工程师 | systematic-debugging（失败用例根因分析） |

## 7. 单人开发流程裁剪指南

本项目为单人开发模式，完整团队流程在以下场景中可合理裁剪：

### 7.1 裁剪原则

1. **方法论不可省** — TDD、systematic-debugging、verification-before-completion 始终执行
2. **文档按需产出** — 根据变更影响面决定文档完整度
3. **角色合并执行** — 同一人连续执行多角色，减少形式化切换

### 7.2 按任务类型裁剪

| 任务类型 | 执行流程 | 文档产出 |
|---------|---------|---------|
| **小优化/Bug修复**（≤半天）| systematic-debugging → TDD → verification | 仅代码+测试，可选 Bug修复报告 |
| **中等功能**（半天~2天）| brainstorming（轻量）→ TDD → verification → 代码评审 | 代码+测试+交付报告 |
| **大功能/架构变更**（>2天）| 完整流程：brainstorming → 需求分析 → 架构设计 → 任务拆解 → TDD → 代码评审 → 测试执行 | 全套文档 |
| **紧急Hotfix** | systematic-debugging（加速Phase 2-3）→ TDD → verification → 回归测试 | 修复代码+测试+回归报告 |

### 7.3 流程步骤合并建议

```
完整链路（12步）：
  产品规划 → 需求分析 → 架构设计 → 架构评审 → 任务拆解 → 
  writing-plans → 功能开发 → 代码评审 → 测试策略 → 测试执行 → 
  回归测试 → 发布

轻量链路（5步，适用大多数迭代）：
  需求确认 → [TDD] 功能开发 → [verification] 代码评审 → 
  测试执行 → 发布
```

**合并规则**：
- 产品规划 + 需求分析 → **需求确认**（一条记录即可）
- 架构设计 + 架构评审 → 仅重大变更时执行
- 任务拆解 + writing-plans → 仅复杂任务保留
- 测试策略 → 仅首次或测试框架变更时
- 文档归档 → 发布时顺带完成

### 7.4 时效规则说明

第6节时效规则定义的是团队协作场景的参考值。单人模式下，按实际优先级自行安排，但 **P0级安全/Bug问题应立即处理**。
