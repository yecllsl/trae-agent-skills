# 完整流程调用规范

> **打开时机**：需要知道"这一步如何调用Skill、下一步调什么"时打开。—— **执行型调用手册**。

本文档定义完整开发流程中每个环节的Skill调用模式，包括全局Skill与项目Skill的嵌入关系、验证规则和异常处理。

***

## 1. 阶段0：产品规划（产品经理主导）

### 1.1 正常流程

```
用户需求/业务目标
   ↓
brainstorming（需求不明确时前置）
   ├─ 逐个提问澄清意图
   ├─ 提出2-3种方案对比
   └─ 输出: 设计文档
   ↓
产品规划（PM-01）
   ├─ 输入: 市场调研、业务目标、brainstorming输出
   ├─ 验证: 产品愿景清晰、路线图合理
   └─ 输出: /docs/product/产品规划方案.md
   ↓
verification-before-completion（守门验证）
   ├─ 运行: 检查产品规划方案完整性
   └─ 输出: 验证证据
```

### 1.2 异常处理

| 异常场景 | 处理方式 | 回退Skill |
|---------|---------|----------|
| 产品方向不明确 | 重新定义产品愿景 | brainstorming |
| 需求理解有歧义 | 逐个提问澄清 | brainstorming（需求澄清模式） |

## 2. 阶段1：需求分析与架构设计（架构师主导）

### 2.1 需求分析

```
产品规划方案
   ↓
brainstorming（需求澄清，可选）
   ├─ 拆解业务诉求
   └─ 输出: 需求澄清记录
   ↓
需求分析（ARC-01）
   ├─ 输入: 产品规划方案、业务诉求
   ├─ 验证: 需求无歧义、验收标准可量化
   └─ 输出: /docs/requirements/REQ_需求规格说明书.md
   ↓
verification-before-completion（守门验证）
   └─ 输出: 验证证据
```

### 2.2 架构设计

```
需求规格说明书
   ↓
架构设计（ARC-02）
   ├─ 输入: 需求规格说明书
   ├─ 验证: 技术栈适配、接口规范完整
   └─ 输出: /docs/architecture/架构设计说明书.md
   ↓
verification-before-completion（守门验证）
   └─ 输出: 验证证据
```

### 2.3 架构评审

```
架构设计方案 + 需求规格
   ↓
架构评审（ARC-04）
   ├─ 输入: 架构设计方案、需求规格
   ├─ 验证: 技术可行性验证、风险识别完整
   └─ 输出: /docs/architecture/架构评审报告.md
   ↓
verification-before-completion（评审结论需验证证据）
   ├─ 运行: 检查评审报告完整性
   └─ 输出: 验证证据
   ↓
[评审不通过] → 按评审报告整改 → 重新提交架构评审
[评审通过] → 进入任务拆解
```

### 2.4 任务拆解

```
架构方案（评审通过后）
   ↓
任务拆解（ARC-03）
   ├─ 输入: 架构设计方案
   ├─ 验证: 任务粒度适配迭代周期、依赖无闭环
   └─ 输出: /docs/planning/task_list_*.md
   ↓
writing-plans（微观展开实施计划）
   ├─ 输入: 任务清单
   ├─ 验证: 每步包含实际内容、非占位符
   └─ 输出: 实施计划文档
   ↓
verification-before-completion（守门验证）
   └─ 输出: 验证证据
```

## 3. 阶段2：开发实现（开发工程师主导）

### 3.1 功能开发（单任务模式）

```
实施计划（writing-plans输出）
   ↓
using-git-worktrees（创建隔离工作区）
   ├─ 输入: 实施计划
   └─ 输出: 隔离worktree配置
   ↓
executing-plans（独立会话批量执行）
   ↓
 Task 1
   ├─ 调用 feature-development 技能
   │   ├─ 内嵌 TDD: 先写失败测试 → 写实现 → 测试通过
   │   ├─ 内嵌 verification: 每步运行pytest/ruff/mypy验证
   │   └─ 输出: 代码 + 测试 + 交付报告片段
   ↓
 Task 2
   ├─ 调用 feature-development 技能
   │   ├─ 内嵌 TDD: 先写失败测试 → 写实现 → 测试通过
   │   ├─ 内嵌 verification: 每步运行pytest/ruff/mypy验证
   │   └─ 输出: 代码 + 测试 + 交付报告片段
   ↓
 ...
   ↓
 所有Task完成
   ↓
verification-before-completion（开发完成验证）
   ├─ 运行: pytest + ruff + mypy
   ├─ 验证: 覆盖率≥80%、测试通过率100%
   └─ 输出: 验证证据
   ↓
功能开发（DEV-01）汇总输出
   ├─ 代码 + /tests/unit/ + /docs/development/交付报告_*.md
   └─ 输出: 完整交付报告
```

### 3.2 功能开发（多任务并行模式）

```
实施计划（writing-plans输出，含2+独立任务）
   ↓
using-git-worktrees（创建隔离工作区）
   ↓
dispatching-parallel-agents（并行调度）
   ├─ Agent A → feature-development: 功能A
   │   ├─ 内嵌 TDD: 先写失败测试 → 写实现 → 测试通过
   │   ├─ 内嵌 verification: 每步运行pytest/ruff/mypy验证
   │   └─ 输出: 代码A + 测试A + 交付报告A
   ├─ Agent B → feature-development: 功能B
   │   ├─ 内嵌 TDD: 先写失败测试 → 写实现 → 测试通过
   │   ├─ 内嵌 verification: 每步运行pytest/ruff/mypy验证
   │   └─ 输出: 代码B + 测试B + 交付报告B
   └─ ...
   ↓
汇总所有Agent输出
   ↓
verification-before-completion（开发完成验证）
   ├─ 运行: pytest + ruff + mypy
   ├─ 验证: 覆盖率≥80%、测试通过率100%
   └─ 输出: 验证证据
   ↓
功能开发（DEV-01）汇总输出
   └─ 完整交付报告
```

### 3.3 功能开发（当前会话分步执行模式）

```
实施计划（writing-plans输出）
   ↓
using-git-worktrees（创建隔离工作区）
   ↓
subagent-driven-development（当前会话分步执行）
   ↓
 Task 1
   ├─ 调用 feature-development 技能
   │   ├─ 内嵌 TDD: 先写失败测试 → 写实现 → 测试通过
   │   ├─ 内嵌 verification: 每步运行pytest/ruff/mypy验证
   │   └─ 输出: 代码 + 测试 + 交付报告片段
   ↓
 Task 2（同上）
   ↓
 ...
   ↓
 所有Task完成
   ↓
verification-before-completion（开发完成验证）
   └─ 输出: 验证证据
   ↓
功能开发（DEV-01）汇总输出
   └─ 完整交付报告
```

### 3.4 代码评审

```
开发交付报告 + 代码仓库 + 架构方案
   ↓
requesting-code-review（可选快检前置）
   ├─ 输入: 代码变更、架构方案
   ├─ 验证: 代码质量（规范、安全、性能）
   └─ 输出: 评审反馈
   ↓
代码评审（REV-01）
   ├─ 输入: 代码仓库、架构方案、开发交付报告
   ├─ 验证: 核心代码无架构偏离、代码规范达标
   └─ 输出: /docs/review/代码评审报告_*.md
   ↓
verification-before-completion（评审结论验证）
   └─ 输出: 验证证据
   ↓
[评审不通过] → receiving-code-review（评估反馈）→ 按评审报告整改 → 重新提交
[评审通过] → 进入测试阶段
```

## 4. 阶段3：测试验证（测试工程师主导）

### 4.1 测试策略

```
需求规格 + 架构方案
   ↓
测试策略（TST-01）
   ├─ 输入: 需求规格、架构方案
   ├─ 验证: 覆盖MVP需求、准入准出可量化
   └─ 输出: /docs/test/strategy_*.md
   ↓
verification-before-completion（守门验证）
   └─ 输出: 验证证据
```

### 4.2 测试执行

```
代码仓库 + 测试用例
   ↓
测试执行（TST-02）
   ├─ 输入: 代码仓库、测试用例
   ├─ 验证: 用例执行率100%、核心流程通过率≥95%
   └─ 输出: /docs/test/测试报告_*.md + /docs/test/Bug清单_*.md
   ↓
[测试通过率≥95%] → 进入Bug修复（针对失败用例）
[测试通过率<95%] → 优先修复失败用例 → 重新测试执行
```

### 4.3 Bug修复

```
Bug清单
   ↓
systematic-debugging（Bug根因不明时前置）
   ├─ Phase 1: 读错误→复现→检查变更→收集证据
   ├─ Phase 2: 找正常案例→对比差异
   ├─ Phase 3: 形成假设→最小测试
   └─ Phase 4: 创建失败测试→修复→验证
   ↓
Bug修复（DEV-02）
   ├─ 输入: Bug清单、根因分析
   ├─ 内嵌 TDD: 先写失败测试 → 写修复 → 测试通过
   ├─ 内嵌 verification: 运行pytest/ruff/mypy验证
   └─ 输出: 修复代码 + /docs/development/Bug修复报告_*.md
   ↓
verification-before-completion（修复验证）
   ├─ 运行: pytest + ruff + mypy
   ├─ 验证: Bug复现验证通过、无新Bug引入
   └─ 输出: 验证证据
   ↓
[修复3次仍失败] → 停止修复 → 质疑架构设计 → 架构师介入
[修复成功] → 进入回归测试
```

### 4.4 回归测试

```
Bug修复报告 + 修复代码
   ↓
回归测试（TST-03）
   ├─ 输入: Bug修复报告、修复代码
   ├─ 验证: P0/P1 Bug 100%验证、无副作用
   └─ 输出: /docs/test/回归报告_*.md + /docs/test/上线结论_*.md
   ↓
verification-before-completion（上线结论需验证证据）
   ├─ 运行: 全量回归测试
   ├─ 验证: P0/P1 Bug 100%修复验证
   └─ 输出: 验证证据
   ↓
[上线结论="建议上线"] → 进入发布阶段
[上线结论="不建议上线"] → 返回Bug修复
```

## 5. 阶段4：自动化发布（发布运维主导）

### 5.1 文档更新

```
上线结论 + 版本号 + 变更内容
   ↓
文档更新（OPS-01）
   ├─ 输入: 上线结论、版本号、变更内容
   ├─ 验证: 文档与版本内容一致
   └─ 输出: 更新后的用户文档 + 全局文档
      ├─ CHANGELOG、README、CLI使用指南
      ├─ API文档、Agent配置指南
      └─ AGENTS.md、架构设计说明书、需求规格说明书等全局指路文档
   ↓
verification-before-completion（文档一致性验证）
   └─ 输出: 验证证据
```

### 5.2 CICD验证

```
Git分支 + 构建命令
   ↓
CICD验证（OPS-02）
   ├─ 输入: Git分支、构建命令
   ├─ 验证: 流水线自动触发、步骤无报错
   └─ 输出: /docs/devops/流水线执行报告_*.md
   ↓
[流水线通过] → 进入自动化发布
[流水线失败] → FIX-CI（OPS-04）→ 重新CICD验证
```

### 5.3 FIX-CI（CI构建失败时）

```
CI构建失败日志
   ↓
FIX-CI（OPS-04）
   ├─ 输入: Git分支、构建命令、失败日志
   ├─ 诊断: 自动获取gh cli日志、分析错误类型
   ├─ 验证: 修复后流水线重新执行通过
   └─ 输出: 修复配置 + /docs/devops/FIX-CI报告_*.md
   ↓
[配置型错误] → 直接修复workflow文件 → 重新CICD验证
[代码型错误] → 停止 → 切换到systematic-debugging → Bug修复 → 重新CICD验证
```

### 5.4 单人自动化发布

```
上线结论（"建议上线"）+ 版本号 + main分支
   ↓
verification-before-completion（最终守门验证）
   ├─ 验证: 发布准入条件全部满足
   │   ├─ 测试上线结论为"建议上线"
   │   ├─ P0/P1级Bug 100%修复验证
   │   ├─ CICD流水线验证通过
   │   ├─ 版本号已更新
   │   ├─ 用户文档已更新
   │   └─ 全局指路文档已更新
   └─ 输出: 验证证据
   ↓
单人自动化发布（OPS-03）
   ├─ 输入: 上线结论、版本号
   ├─ 内嵌 git-pushing: 版本号更新、Tag创建、推送发布
   ├─ 验证: Tag创建成功、线上服务正常
   └─ 输出: 版本Tag + /docs/devops/发布报告_*.md
   ↓
verification-before-completion（发布完成验证）
   ├─ 运行: 检查Tag创建成功、Release发布成功
   └─ 输出: 验证证据
```

### 5.5 文档归档

```
版本号 + 发布报告
   ↓
文档归档（OPS-05）
   ├─ 输入: 版本号、发布报告
   ├─ 验证: 版本文档已归档、项目目录整洁
   └─ 输出: 归档目录 + 清理后的项目目录
      ├─ 归档: 测试报告、发布报告、开发报告、评审报告
      └─ 清理: docs/test/reports/、docs/devops/、docs/development/、docs/review/
   ↓
verification-before-completion（归档完成验证）
   └─ 输出: 验证证据
```

## 6. 集成决策：finishing-a-development-branch

### 6.1 触发时机

所有开发Task完成后，进入集成决策：

```
所有Task完成 + verification-before-completion验证通过
   ↓
finishing-a-development-branch
   ├─ 选项1: 合并到main分支（推荐）
   │   ├─ 运行: 全量测试验证
   │   └─ 执行: git merge
   ├─ 选项2: 创建Pull Request
   │   ├─ 运行: 全量测试验证
   │   └─ 执行: git push + gh pr create
   └─ 选项3: 清理worktree（暂不合并）
      └─ 执行: git worktree remove
   ↓
[合并/PR完成] → 进入代码评审 → 测试验证 → 发布
[清理worktree] → 保留分支，等待后续处理
```

## 7. 异常处理流程

### 7.1 开发阶段异常

```
[开发过程中遇到问题]
   ↓
┌─ 需求不明确 → brainstorming（需求澄清）
├─ 架构设计疑问 → 查阅架构设计说明书 → 必要时架构师介入
├─ 本地调试多次失败 → systematic-debugging（根因定位）
│   └─ 输出: 调试报告 → 同步架构师排查
├─ TDD测试失败 → systematic-debugging（Phase 1-3根因定位）
└─ 代码评审不通过 → receiving-code-review（评估反馈）→ 整改 → 重新提交
```

### 7.2 测试阶段异常

```
[测试过程中遇到问题]
   ↓
┌─ Bug根因不明 → systematic-debugging（四阶段根因定位）
├─ Bug修复3次仍失败 → 停止修复 → 质疑架构设计 → 架构师介入
├─ 测试通过率<95% → 优先修复失败用例 → 重新测试
└─ 回归测试发现新Bug → 返回Bug修复流程
```

### 7.3 发布阶段异常

```
[发布过程中遇到问题]
   ↓
┌─ CICD流水线失败 → FIX-CI（配置型直接修复，代码型转systematic-debugging）
├─ CICD连续失败 → 同步架构师、测试工程师共同分析
├─ 发布失败需回滚 → 单人自动化发布中的回滚指引
└─ 版本Tag创建失败 → git-pushing（错误排查）→ 重新发布
```

## 8. 单人开发流程裁剪

### 8.1 轻量链路（适用大多数迭代）

```
需求确认（产品规划+需求分析合并）
   ↓
brainstorming（轻量需求澄清，可选）
   ↓
writing-plans（直接编写实施计划，跳过任务拆解）
   ↓
using-git-worktrees（创建隔离工作区）
   ↓
executing-plans / subagent-driven-development
   ├─ 每个Task内嵌: TDD + verification
   └─ 输出: 代码 + 测试 + 交付报告
   ↓
verification-before-completion（开发完成验证）
   ↓
代码评审（REV-01）
   ↓
测试执行（TST-02，跳过测试策略）
   ↓
[Bug] → systematic-debugging → Bug修复 → verification
   ↓
回归测试（TST-03）
   ↓
verification-before-completion（上线前验证）
   ↓
文档更新（OPS-01）→ CICD验证（OPS-02）→ 单人自动化发布（OPS-03）→ 文档归档（OPS-05）
```

### 8.2 小优化/Bug修复（≤半天）

```
systematic-debugging（根因定位）
   ↓
Bug修复（DEV-02）
   ├─ 内嵌 TDD: 先写失败测试 → 写修复 → 测试通过
   └─ 内嵌 verification: 运行pytest/ruff/mypy验证
   ↓
verification-before-completion（修复验证）
   ↓
[可选] 回归测试 → 发布
```

### 8.3 紧急Hotfix

```
systematic-debugging（加速Phase 2-3）
   ↓
Bug修复（DEV-02）
   ├─ 内嵌 TDD: 先写失败测试 → 写修复 → 测试通过
   └─ 内嵌 verification: 运行pytest/ruff/mypy验证
   ↓
回归测试（TST-03，最小回归验证）
   ↓
单人自动化发布（OPS-03，紧急发布，修订版本号递增）
```

## 9. 全局Skill铁律速查

| Skill | 铁律 | 违反后果 |
|-------|------|---------|
| brainstorming | 不经设计批准，不写任何代码 | 方向错误、返工 |
| test-driven-development | 没有失败测试，不写生产代码 | 测试覆盖不足 |
| systematic-debugging | 没有根因调查，不提任何修复 | 修复无效、引入新Bug |
| verification-before-completion | 没有新鲜验证证据，不做完成声称 | 虚假完成、质量失控 |
| writing-plans | 每步必须包含实际内容，禁止占位符 | 计划无法执行 |
