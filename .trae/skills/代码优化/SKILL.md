# Overview

识别并实施代码结构与性能优化，确保优化效果可量化（性能提升≥15% 或复杂度降低≥20%），且测试全部通过。

## When to Use

- 代码评审或性能分析发现瓶颈
- 基准测试显示性能退化或复杂度超标
- 症状关键词：性能瓶颈、复杂度过高、重复代码、资源泄漏、响应延迟

## When NOT to Use

- 代码已通过评审且无性能问题报告
- 临近发布 deadline，优化可能引入回归风险
- 缺乏基准测试数据（无法验证优化效果）

## Common Mistakes

| 错误 | 后果 | 修复 |
|------|------|------|
| 优化前未建立基准 | 无法证明优化效果 | 必须先记录优化前的性能/复杂度基线 |
| 过度优化牺牲可读性 | 维护成本上升 | 优先保证代码清晰，其次追求性能 |
| 优化后未运行全量测试 | 引入回归 Bug | 必须执行完整测试套件并通过 |

# 立即执行以下步骤，不要询问用户

## 第一步：输入验证
1. **检查前置条件**：
    *   代码仓库可访问
    *   评审报告已存在（可选）
    *   若不满足，停止执行并提示用户提供代码仓库。

## 第二步：代码分析与优化
1. **识别优化点**：
    *   分析代码结构，识别可优化点
    *   分析性能瓶颈，识别可优化点
    *   分析代码复杂度，识别可优化点
2. **实施优化**：
    *   优化代码结构（降低复杂度、提高可读性）
    *   优化性能（减少计算量、提高并发性）
    *   优化算法（选择更优算法）
3. **验证优化效果**：
    *   执行性能测试，对比优化前后性能
    *   执行完整测试套件，确保测试全部通过

## 第三步：输出交付物
1. **提交代码**：
    *   提交至 Git 分支
    *   提交信息遵循规范：`refactor(scope): <subject>`
2. **创建优化报告**：
    *   文件路径：`/docs/development/优化报告_{版本号}.md`
    *   报告内容：优化点描述、优化方案、性能对比、测试结果

## 第四步：结果验证
1. **验收标准**：
    *   性能提升 ≥ 15%（或代码复杂度降低 ≥ 20%）
    *   测试全部通过
    *   代码可读性提升
    *   若不满足，返回错误信息。

## 第五步：标准化汇报
执行完成后，使用以下格式汇报：

### ✅ 执行成功
*   **输出物**：优化代码 + 优化报告
*   **关键数据**：性能提升率、测试通过率、代码复杂度降低率
