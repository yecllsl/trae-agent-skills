# 全局技能与项目技能协作使用指导规范

> **打开时机**：需要做"该用哪个Skill"的决策时打开。—— **决策型方法论手册**。流程步骤请查阅 [协作链路.md](./协作链路.md)，指令定义请查阅 [指令手册.md](./指令手册.md)。

本文档定义全局Skills与项目Skills的协作模式、决策规则和场景化最佳实践，确保两类技能各司其职、协同增效。

***

## 1. 核心原则

### 1.1 两类技能的本质定位

| 维度 | 全局Skills | 项目Skills |
|------|-----------|-----------|
| **本质** | 方法论（How to think） | 执行器（How to do） |
| **范围** | 跨项目通用，与具体业务无关 | 绑定当前项目，包含业务知识 |
| **输出** | 设计文档、计划、验证证据 | 标准化交付物（代码+报告） |
| **角色** | 思考框架、质量守门员 | 角色扮演、流程标准化 |
| **耦合度** | 零耦合，可迁移到任何项目 | 高耦合，依赖项目文档结构 |

### 1.2 协作黄金法则

1. **方法论先行** — 全局Skill决定"怎么想"，项目Skill决定"怎么做"，思考先于执行
2. **串联叠加** — 两类Skill是串联叠加关系，不是互斥替代关系
3. **跨切面嵌入** — verification-before-completion、TDD等全局Skill叠加在所有项目Skill之上
4. **准入门槛** — 项目Skill有前置条件，不满足时先回退到上游全局Skill补齐
5. **紧急不跳方法论** — 紧急场景可压缩流程，但方法论铁律（如debugging的根因定位）不可违反

### 1.3 调用优先级

```
用户指令 → 检查是否有相关Skill → 先检查全局Skill（方法论）→ 再调用项目Skill（执行）
```

当多个Skill可能适用时，遵循以下优先级：

1. **流程型全局Skill**（brainstorming、systematic-debugging）— 决定如何切入问题
2. **实施型全局Skill**（writing-plans、TDD）— 指导如何执行
3. **项目Skill** — 按标准化流程交付
4. **质量守门型全局Skill**（verification-before-completion）— 最终验证

## 3. 功能重叠决策矩阵

### 3.1 需求探索阶段

| 对比维度 | brainstorming（全局） | 产品规划（项目） | 需求分析（项目） |
|----------|----------------------|-----------------|-----------------|
| **目的** | 开放性探索，理解意图 | 定义产品愿景和路线图 | 拆解功能点、定义验收标准 |
| **交互方式** | 逐个提问、多方案对比 | 标准化输出 | 标准化输出 |
| **输出** | 设计文档（自由格式） | 产品规划方案.md | REQ_需求规格说明书.md |
| **适用时机** | 最早介入 | 有明确业务目标后 | 有产品规划后 |

**决策规则**：三者串联，非互斥。brainstorming → 产品规划 → 需求分析。

- 需求不明确 → 先brainstorming
- 需求明确但缺产品规划 → 产品规划
- 有产品规划但缺需求规格 → 需求分析

### 3.2 任务规划阶段

| 对比维度 | writing-plans（全局） | 任务拆解（项目） |
|----------|----------------------|-----------------|
| **粒度** | 5-15分钟/步，可执行到函数/文件级 | 1-3天/任务，模块级 |
| **内容** | 具体操作+预期结果+验证方式 | 任务概述+优先级+依赖+工作量估算 |
| **受众** | 执行者（人或Agent） | 项目管理者 |
| **TDD** | 每步嵌入RED-GREEN-REFACTOR循环 | 仅标注"需测试"标记，不展开具体步骤 |

**决策规则**：按任务规模裁剪，避免流程冗余。

| 任务规模 | 策略 |
|----------|------|
| 小任务（< 半天） | 仅 writing-plans，无需任务拆解 |
| 中任务（半天~2天） | writing-plans 为主，任务拆解做简要记录 |
| 大任务（> 2天） | 两者都用，任务拆解用于里程碑追踪 |

**执行顺序**：

- 正向开发（新需求）：任务拆解（宏观切分）→ writing-plans（微观展开）
- 复盘归档（已完成）：writing-plans（执行记录）→ 任务拆解（里程碑清单）

### 3.3 开发方法论

| 对比维度 | test-driven-development（全局） | 功能开发（项目） |
|----------|-------------------------------|-----------------|
| **性质** | 方法论（怎么写代码） | 执行框架（按什么流程交付） |
| **覆盖范围** | 仅关注编码阶段的测试循环 | 覆盖输入验证→编码→测试→交付→验证全流程 |
| **输出** | 无独立输出（嵌入开发过程） | 代码+测试+交付报告 |

**决策规则**：TDD嵌入功能开发使用，不是替代关系。

- 功能开发定义"做什么、交付什么"
- TDD定义"怎么写每一行代码"
- 优先级：功能开发是主流程，TDD是子流程

### 3.4 问题定位

| 对比维度 | systematic-debugging（全局） | Bug修复（项目） |
|----------|----------------------------|-----------------|
| **关注点** | 根因定位（Phase 1-3） | 修复执行（第二步起） |
| **输出** | 根因分析（无独立文档） | Bug修复报告.md |

**决策规则**：

- Bug根因未知 → 先systematic-debugging
- Bug根因已知（如测试报告已定位） → 直接Bug修复
- 3次修复失败 → systematic-debugging质疑架构

### 3.5 代码评审

| 对比维度 | requesting-code-review（全局） | 代码评审（项目） |
|----------|-------------------------------|-----------------|
| **方式** | 调度子代理评审 | 主Agent直接评审 |
| **视角** | 代码质量（规范、安全、性能） | 架构一致性+代码质量+安全审计 |
| **上下文** | 隔离上下文，独立评审 | 需要架构方案和交付报告 |
| **适用** | Subagent-driven开发模式 | 项目标准流程 |

**决策规则**：

- 使用subagent-driven-development → requesting-code-review（每任务后评审）
- 使用项目标准流程 → 代码评审（含架构一致性检查）
- 两者可组合：先用requesting-code-review做代码质量快检，再用代码评审做架构一致性深审

### 3.6 完成验证

| 对比维度 | verification-before-completion（全局） | 项目Skill的验证步骤 |
|----------|--------------------------------------|-------------------|
| **性质** | 元规则（声称完成前必须有证据） | 流程步骤（检查验收标准） |
| **强度** | 铁律，不可违反 | 可根据情况调整 |
| **范围** | 适用于所有场景 | 仅适用于当前Skill |

**决策规则**：verification-before-completion是跨切面关注点，叠加在所有项目Skill之上。

- 项目Skill的验证步骤是"检查清单"
- verification是"证据要求"
- 每次声称完成时，都必须用verification的Gate Function

### 3.7 Git操作

| 对比维度 | git-pushing（全局） | 单人自动化发布（项目） |
|----------|-------------------|---------------------|
| **范围** | 通用Git操作参考 | 项目特定的发布流程 |
| **深度** | 覆盖所有Git场景 | 聚焦版本发布场景 |
| **项目知识** | 无 | 包含版本号一致性检查、CI验证顺序 |

**决策规则**：

- 版本发布 → 单人自动化发布（已内嵌Git操作）
- 日常开发中的Git操作 → git-pushing
- 发布失败需回滚 → 单人自动化发布中的回滚指引

## 4. 场景化实施步骤

### 4.1 场景一：新功能开发（从零到交付）

**触发条件**：用户提出全新功能需求

```
阶段1: 需求澄清
  ① brainstorming（全局）→ 探索意图、提问澄清、设计方案
  ② 产品规划（项目）→ 若涉及版本路线图调整
  ③ 需求分析（项目）→ 拆解功能点、定义验收标准

阶段2: 架构与规划
  ④ 架构设计（项目）→ 技术选型、模块划分、接口设计
  ⑤ 架构评审（项目）→ 验证技术可行性
  ⑥ 任务拆解（项目）→ 宏观切分任务
  ⑦ writing-plans（全局）→ 微观展开实施计划

阶段3: 开发实施
  ⑧ using-git-worktrees（全局）→ 创建隔离工作区
  ⑨ 功能开发（项目）→ 按任务清单编码
     内嵌：test-driven-development（全局）→ TDD循环
     内嵌：verification-before-completion（全局）→ 每步验证

阶段4: 质量保障
  ⑩ 代码评审（项目）→ 架构一致性+安全审计
  ⑪ 测试策略（项目）→ 制定门禁规则
  ⑫ 测试执行（项目）→ 执行测试、输出Bug清单

阶段5: 收尾
  ⑬ finishing-a-development-branch（全局）→ 合并/PR
  ⑭ verification-before-completion（全局）→ 最终验证
```

### 4.2 场景二：Bug修复

**触发条件**：测试报告或用户反馈Bug

```
阶段1: 问题定位
  ① systematic-debugging（全局）→ 四阶段根因定位
     Phase 1: 读错误→复现→检查变更→收集证据
     Phase 2: 找正常案例→对比差异
     Phase 3: 形成假设→最小测试
     Phase 4: 创建失败测试→修复→验证

阶段2: 修复实施
  ② Bug修复（项目）→ 按项目规范执行修复流程
     内嵌：test-driven-development（全局）→ 先写失败测试
     内嵌：verification-before-completion（全局）→ 验证修复

阶段3: 验证闭环
  ③ 回归测试（项目）→ 验证无副作用、给出上线结论
```

### 4.3 场景三：代码重构/优化

**触发条件**：代码评审或性能分析发现需要重构

```
阶段1: 设计
  ① brainstorming（全局）→ 探索重构方案、权衡取舍

阶段2: 实施
  ② 代码优化（项目）→ 按项目规范执行优化
     内嵌：test-driven-development（全局）→ 重构前先有测试保护
     内嵌：verification-before-completion（全局）→ 性能对比验证

阶段3: 验证
  ③ 代码评审（项目）→ 检查优化是否引入问题
  ④ 测试执行（项目）→ 全量回归
```

### 4.4 场景四：CI/CD流水线故障

**触发条件**：GitHub Actions构建失败

```
  ① FIX-CI（项目）→ 自动获取日志、诊断、修复workflow配置
     若判断为配置型错误 → 直接修复
     若判断为代码型错误 → 停止，提示修复业务代码

  ② 若代码型错误 → 切换到 systematic-debugging（全局）
     → 定位代码根因 → Bug修复（项目）

  ③ CICD验证（项目）→ 修复后验证流水线
```

### 4.5 场景五：版本发布

**触发条件**：所有开发测试完成，准备发布新版本

```
  ① verification-before-completion（全局）→ 最终验证所有检查项
  ② 文档更新（项目）→ 更新CHANGELOG/README/API文档/全局文档
  ③ CICD验证（项目）→ 验证流水线通过
  ④ 单人自动化发布（项目）→ 版本号更新、Tag创建、推送发布
  ⑤ 文档归档（项目）→ 归档版本文档、清理目录
```

### 4.6 场景六：紧急Hotfix

**触发条件**：线上发现严重Bug，需紧急修复发布

```
  ① systematic-debugging（全局）→ 快速定位根因
     （紧急情况下仍须遵循Phase 1，但可加速Phase 2-3）
  ② Bug修复（项目）→ 修复+测试
  ③ 回归测试（项目）→ 最小回归验证
  ④ 单人自动化发布（项目）→ 紧急发布（修订版本号递增）
```

**注意**：紧急场景下，全局Skill的方法论不可省略（跳过debugging直接修复是最大风险），但流程可压缩（文档归档可延后）。

### 4.7 场景七：探索性任务/技术调研

**触发条件**：不确定要做什么，需要先探索

```
  ① brainstorming（全局）→ 开放性探索
     逐个提问澄清 → 提出2-3种方案 → 用户选择
  ② 若范围过大 → brainstorming帮助分解为子项目
  ③ 确定方向后 → 进入场景一的流程
```

**注意**：此场景仅使用全局Skill，项目Skill在需求明确前不介入。

### 4.8 场景八：多任务并行开发

**触发条件**：有多个独立任务需要并行推进

```
  ① brainstorming（全局）→ 分别设计各功能
  ② writing-plans（全局）→ 分别编写实施计划
  ③ dispatching-parallel-agents（全局）→ 并行调度子代理
     Agent A → 功能开发（项目）：功能A
     Agent B → 功能开发（项目）：功能B
  ④ 各Agent内部遵循：TDD + verification
  ⑤ 汇总后 → 代码评审（项目）→ 测试执行（项目）
```

## 5. 五维决策框架

当面临Skill选择时，按以下5个维度评估：

| 维度 | 选全局Skill的信号 | 选项目Skill的信号 |
|------|-----------------|-----------------|
| **1. 需求性质** | 需求模糊、需探索 | 需求明确、需执行 |
| **2. 输出要求** | 需要思考过程、方案对比 | 需要标准化文档、合规交付 |
| **3. 项目依赖** | 不依赖项目文档结构 | 依赖项目文档链路（准入/准出） |
| **4. 质量要求** | 需要方法论保障（防遗漏/防偏差） | 需要流程保障（防跳步/防不规范） |
| **5. 团队协作** | 个人思考、设计决策 | 团队对齐、交付验收 |

**快速决策口诀**：

- 想不清楚 → 全局Skill（brainstorming / debugging）
- 知道做什么但不知道怎么做 → 全局Skill（writing-plans / TDD）
- 知道怎么做且需要交付 → 项目Skill
- 声称完成 → 全局Skill（verification-before-completion）

## 6. 注意事项

### 6.1 常见反模式

| 反模式 | 描述 | 正确做法 |
|--------|------|---------|
| **跳过探索直接执行** | 需求不明确时直接调用项目Skill | 先brainstorming澄清需求 |
| **跳过debugging直接修复** | Bug根因未明时直接修改代码 | 先systematic-debugging定位根因 |
| **跳过TDD直接编码** | 先写实现再补测试 | 先写失败测试再写实现 |
| **跳过verification声称完成** | "应该没问题"就宣布完成 | 运行验证命令，拿证据再声称 |
| **重叠Skill二选一** | brainstorming和产品规划只选一个 | 两者串联使用 |
| **项目Skill无准入执行** | 不检查前置条件直接执行 | 先验证前置条件，不满足则回退 |
| **紧急场景跳过方法论** | 紧急时跳过根因分析 | 方法论不可省略，流程可压缩 |
| **只用全局Skill不产出交付物** | 停留在方法论层面，未落地 | 全局Skill是方法论，最终通过项目Skill产出标准化交付物 |

### 6.2 项目Skill准入条件速查

| 项目Skill | 必须满足的前置条件 |
|-----------|------------------|
| 产品规划 | 市场调研或用户反馈已提供 |
| 需求分析 | 产品规划方案已存在 |
| 架构设计 | 需求规格说明书已存在 |
| 架构评审 | 架构设计方案+需求规格已存在 |
| 任务拆解 | 架构设计方案已存在 |
| 功能开发 | 任务清单+架构方案已存在 |
| Bug修复 | Bug清单已存在 |
| 代码优化 | 代码仓库可访问 |
| 代码评审 | 代码新提交+架构方案+交付报告已存在 |
| 测试策略 | 需求规格+架构方案已存在 |
| 测试执行 | 代码仓库+测试用例已存在 |
| 回归测试 | Bug修复报告已存在 |
| CICD验证 | Git分支+构建命令已提供 |
| 单人自动化发布 | 回归报告+版本号+main分支 |
| 文档更新 | 版本号+变更内容已确认 |
| 文档归档 | 版本已发布（Tag+Release） |

### 6.3 全局Skill铁律

| Skill | 铁律 |
|-------|------|
| brainstorming | 不经设计批准，不写任何代码 |
| test-driven-development | 没有失败测试，不写生产代码 |
| systematic-debugging | 没有根因调查，不提任何修复 |
| verification-before-completion | 没有新鲜验证证据，不做完成声称 |
| writing-plans | 每步必须包含实际内容，禁止占位符 |

## 7. 最佳实践

### 7.1 协作调用模式

**模式一：串联模式**（最常见）

全局Skill的输出作为项目Skill的输入，形成链式调用。

```
brainstorming → 需求分析 → 架构设计 → writing-plans → 功能开发
```

**模式二：嵌入模式**

全局Skill作为方法论嵌入项目Skill的执行过程中。

```
功能开发 {
    步骤1: 输入验证（项目规范）
    步骤2: TDD循环（全局方法论嵌入）
    步骤3: verification（全局质量门嵌入）
    步骤4: 输出交付物（项目规范）
}
```

**模式三：守门模式**

全局Skill作为质量关卡，在关键节点拦截。

```
功能开发 → [verification-before-completion] → 代码评审
Bug修复 → [verification-before-completion] → 回归测试
全部完成 → [verification-before-completion] → 版本发布
```

**模式四：并行模式**

全局Skill协调多个项目Skill的并行执行。

```
dispatching-parallel-agents → {
    Agent A: 功能开发（Skill A）
    Agent B: 功能开发（Skill B）
}
```

## 8. 决策流程速查

```
用户需求
  │
  ├─ 需求不明确？ ──→ brainstorming（全局）
  │
  ├─ 需求明确，属于哪个阶段？
  │    ├─ 规划阶段 → 产品规划（项目）
  │    ├─ 需求阶段 → 需求分析（项目）
  │    ├─ 架构阶段 → 架构设计（项目）→ 架构评审（项目）
  │    ├─ 开发阶段 → 任务复杂度？
  │    │    ├─ 多任务并行 → dispatching-parallel-agents（全局）
  │    │    ├─ 单任务复杂 → subagent-driven-development（全局）
  │    │    └─ 单任务简单 → 功能开发（项目）
  │    ├─ 测试阶段 → 测试策略（项目）→ 测试执行（项目）
  │    ├─ 发布阶段 → 文档更新→CICD验证→单人自动化发布→文档归档（项目）
  │    └─ 故障阶段 → 故障类型？
  │         ├─ CI配置问题 → FIX-CI（项目）
  │         └─ 代码Bug → systematic-debugging（全局）→ Bug修复（项目）
  │
  └─ 任何声称完成时 → verification-before-completion（全局）
```
